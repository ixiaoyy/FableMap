# Contributing to FableMap

## 项目宣言

FableMap 欢迎协作者，但本项目的协作方式不是“大家各写一套”，而是：

> 先统一世界协议，再并行开发逻辑、内容和表现层。

我们要做的不只是一个游戏原型，而是一套**现实世界到平行世界的映射协议**。

## 协同总原则

- **Protocol-First**：先协议，再实现，再表现
- **工具自由，结果统一**：每个人可以使用不同 AI 或不同工作方式，但提交结果必须符合统一协议
- **局部贡献不得污染全局**：区域内容只能补充与细化，不能偷偷改写全局规则
- **可验证、可回溯、可合并**：所有贡献都必须能说明依据、边界和验证结果
- **小步提交**：一个 PR 尽量只做一类事

## 当前必须遵守的源头文档

在更细的协议文件落地前，当前协作至少以这些文档为准：

- `README.md`
- `docs/WORLD_SCHEMA.md`
- `docs/DISTURBANCE_MODEL.md`
- `docs/FACTION_SYSTEM.md`
- `docs/AESTHETIC_EMOTION_SYSTEMS.md`
- `docs/AI参与开发协议.md`

如果 AI 输出、个人理解或局部灵感与这些文档冲突，**以协议文档为准**。

## 模块认领方式

不要按传统前端/后端硬拆，优先按“现实世界 -> 协议 -> 转义 -> 表现”的逻辑链协作：

### M1：协议与世界观

负责 Schema、映射标准、命名、版本兼容和世界观一致性。

### M2：地理与空间索引

负责 OSM/GIS 清洗、空间切片、区域标识与数据标准化输入。

### M3：转义引擎

负责现实标签到幻想语义的映射、规则实现和 AI 输出约束。

### M4：区域内容

负责具体区域的本地化设定、地点细化、局部风格增强和样例数据。

### M5：客户端表现

负责 Godot / Web / 可视化端的渲染、交互和资产接入。

### M6：验证与开发工具

负责协议校验、测试脚本、预览工具、导出工具和工程化支持。

## 空间协作标准

区域协作的长期标准统一为 **H3 网格**。

在 H3 相关工具完全落地前，任何区域内容贡献至少应在 PR 或文档中声明：

- `h3_index`
- `resolution`
- `region_name`
- `change_type`

目标是避免“我改了东京某片区域”这种模糊描述，统一转成可索引、可合并、可审查的空间单元。

## 贡献类型

### 1. 协议变更

包括：

- Schema 字段调整
- 命名规范变更
- 映射标准变更
- H3 组织方式变更

这类改动必须先说明设计理由和兼容性影响。

### 2. 功能变更

包括：

- CLI / 引擎 / 工具实现
- 数据处理流程
- 客户端表现逻辑
- 测试与验证脚本

### 3. 区域内容变更

包括：

- 某个 H3 区块的局部配置
- 示例世界数据
- 某类地点的本地化补充

区域内容不能擅自修改全局协议。

## 推荐工作流

1. 选择模块或区域任务
2. 确认是否涉及协议变更
3. 新建分支
4. 在明确边界内开发
5. 为本次改动补充对应文档说明
6. 完成最小验证
7. 提交 PR 并附上完整说明

建议分支示例：

- `feature/schema-v0-2-draft`
- `feature/overpass-loader`
- `region/shibuya-8a2a1072b59ffff`
- `docs/ai-dev-protocol`

## Commit 文档要求

从现在开始，**每一次 commit 都必须附带文档说明本次做了什么事情**。

这条规则适用于：

- 代码提交
- 文档提交
- 协议提交
- 区域内容提交
- 测试与工具提交

可接受的形式包括：

- 直接更新相关设计文档
- 直接更新相关模块说明文档
- 为本次改动补一份专门的说明文档

仅写 commit message 不算完成这条要求。

每次 commit 附带的文档说明，至少应包含：

- 本次做了什么
- 为什么要做
- 影响了哪些文件或模块
- 明确没有改什么
- 是否涉及协议 / Schema / 命名变更
- 做了哪些验证

目标是让任何人在只看仓库内容时，也能知道每次提交到底完成了什么，而不是只能靠猜测 diff。

## PR 必填信息

每个 PR 至少要说明：

- 这是哪类改动：协议 / 功能 / 区域内容 / 文档 / 测试
- 涉及哪个模块：M1 ~ M6
- 是否有 AI 参与
- 依据了哪些协议文档
- 修改了哪些文件
- 明确没有修改哪些边界外文件
- 做了哪些验证
- 是否影响兼容性或全局规则

## 验证要求

按改动范围选择最小可行验证，但不能跳过验证。

常见验证包括：

- Schema / JSON / YAML 合法性检查
- 单元测试
- CLI 烟雾测试
- 预览加载检查
- 文档一致性检查

## 对 AI 参与开发的要求

本项目允许任何 AI 参与开发，但不允许任何 AI 绕过协议。

如果你使用 AI，请先阅读：

- `docs/AI参与开发协议.md`

AI 可以辅助分析、草拟、实现、补测试、补文档；
但最终责任始终由提交者本人承担。

## 当前最有价值的贡献方向

- 更细的现实标签 -> 幻想语义映射规则
- 示例区域与 `world.json` 样本
- H3 区域组织与标识约定
- 世界记忆与协议版本化设计
- Godot 占位表现与 JSON 消费链路
- 协议验证器与开发工具

## 禁止事项

- 禁止先写实现、后补协议
- 禁止在功能 PR 中夹带协议修改
- 禁止 AI 擅自新增字段、目录或枚举
- 禁止提交未验证的 AI 产出
- 禁止把局部地区设定上升为全局规范
- 禁止修改任务边界外文件而不说明
