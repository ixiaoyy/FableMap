# brainstorm: 领地系统 - 酒馆申领地块、开店关联真实坐标

> 酒馆申领地块、开店，关联真实坐标，方圆大小被占用，地图充满各式内容

## 用户愿景

- 酒馆可以申领一块地
- 可以开一家店
- 所有领地都关联真实坐标
- 领地有方圆大小，占用空间
- 最终地图上充满各式内容
- 视觉上希望摆脱当前项目偏灰、偏暗的观感，参考「蓝白高饱和科技感 + 二次元引导角色 + 大标题强 CTA + 模块化功能卡片」的明亮商业化风格，让功能入口更有吸引力

---

## 视觉参考与调性方向

用户提供的参考图方向：

- **整体观感**：蓝白为主、高亮、高饱和、轻科幻/云端部署感，避免大面积灰黑底和低对比 UI。
- **信息组织**：首屏使用强标题、强 CTA、右侧/侧边引导角色或视觉焦点，下面用模块化卡片承接功能卖点。
- **功能卡片**：用清晰图标 + 短标题 + 一句话说明表达能力，适合转译到「我的酒馆 / 领地管理 / 一键开店 / 领地申领」入口。
- **情绪目标**：从“后台管理工具”转向“轻松开一间真实地图酒馆/领地”的兴奋感、探索感和拥有感。

落到 FableMap / 领地系统时的约束：

- 仍必须强调**真实坐标锚点**，不能变成无锚点的自由空间。
- 可以使用角色/吉祥物做引导，但不能暗示平台自动替店主生成酒馆内容；空间、NPC、氛围仍由店主确认与配置。
- 视觉提亮应优先服务主链路：选择坐标 → 申领领地 → 开店/配置酒馆 → 对外开放。
- 领地地图层不宜过度游戏化成战争/争夺界面；保留轻经营、轻社区、地图探索的产品语气。

## 核心概念：领地系统 (Territory System)

### 领地定义

领地 = 真实坐标 + 类型 + 半径范围

```
领地 {
  id: string
  owner_id: string
  type: TerritoryType
  center: { lat: number, lon: number }
  radius: number  // 单位：米
  status: TerritoryStatus
  created_at: datetime
}
```

### 领地类型 (TerritoryType)

| 类型 | 图标 | 默认半径 | 功能 |
|------|------|----------|------|
| tavern | 🍺 | 50m | NPC对话、社交 |
| pet_shop | 🐱 | 50m | 养宠互动 |
| garden | 🌱 | 100m | 种菜偷菜 |
| game_workshop | 🎮 | 50m | 小游戏 |
| gacha | 🎰 | 50m | 角色抽卡 |
| cultivation | 🏔️ | 80m | 修行挂机 |
| shop | 🏪 | 30m | 商品交易 |
| warehouse | 📦 | 20m | 存储 |

### 领地大小等级

| 等级 | 半径范围 | 适用类型 |
|------|----------|----------|
| 微型 (tiny) | 10-20m | 摊位、信箱、公告牌 |
| 小型 (small) | 20-40m | 商店、仓库、宠物窝 |
| 中型 (medium) | 40-80m | 酒馆、宠物店、游戏工坊 |
| 大型 (large) | 80-150m | 菜园、加工坊 |
| 巨型 (huge) | 150m+ | 大型酒馆、联盟领地 |

## 坐标占用规则

### 规则定义

```
1. 领地碰撞检测 = 两圆是否相交
   collision = distance(center1, center2) < (radius1 + radius2)

2. 同类型领地不可重叠
   同 type 的两个领地不能同时 active，不能碰撞检测为 true

3. 不同类型领地可以重叠
   酒馆 + 菜园 可以在同一位置（功能分区）

4. 领地可扩展/收缩
   店主操作领地大小，需重新检测碰撞

5. 领地边界可视化
   地图上显示领地范围（半透明色块）
```

### 碰撞检测示例

```
酒馆A (type=tavern, center=116.4, 39.9, radius=50m)
酒馆B (type=tavern, center=116.4, 39.9, radius=50m)
→ 碰撞！不能同时存在

酒馆A (type=tavern, center=116.4, 39.9, radius=50m)
菜园C (type=garden, center=116.4, 39.9, radius=100m)
→ 不碰撞！可以重叠（不同类型）
```

## 领地申领流程

### 流程图

```
用户选择地图坐标
    ↓
系统检测该位置是否已有同类领地
    ↓
┌── 可用 ──┐    ┌── 不可用 ──┐
↓              ↓
显示可申领的领地类型     提示"此处已有同类领地"
    ↓              ↓
选择领地类型         建议：换个位置 / 等该领地废弃
+ 设置领地大小
    ↓
确认申领 → 创建领地
    ↓
设置酒馆内容（NPC、主题等）
    ↓
开放领地
```

### API 设计

```typescript
// 检测位置是否可用
GET /api/v1/territory/check?lat=...&lon=...&radius=...&type=...

// 申领领地
POST /api/v1/territory/claim
Body: {
  type: TerritoryType,
  center: { lat, lon },
  radius: number
}

// 更新领地
PUT /api/v1/territory/{id}
Body: { radius?, status? }

// 获取领地详情
GET /api/v1/territory/{id}

// 列出附近领地
GET /api/v1/territory/nearby?lat=...&lon=...&radius=...&types=...
```

## 领地状态机

```
available → claimed → active → paused → abandoned
              ↓           ↓
              └───────────┴───→ (可恢复)
```

| 状态 | 说明 | 允许操作 |
|------|------|----------|
| available | 可申领 | 无 |
| claimed | 已申领（建设中） | complete, abandon |
| active | 已开放 | pause, resize, update |
| paused | 暂停营业 | resume, abandon |
| abandoned | 已废弃 | 无（可被系统回收） |

## 领地与酒馆关系

- 每个酒馆 = 一个领地
- 酒馆类型决定领地功能
- 领地大小影响酒馆容量
- 领地位置决定地图可见性

### 数据模型

```typescript
// Tavern 扩展领地字段
interface Tavern {
  // 现有字段...
  
  // 新增领地字段

    id: string
    radius: number
    type: TerritoryType
  }
}
```
