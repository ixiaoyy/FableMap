# FableMap 下一阶段规划判断（供 AI 协作者参考）

## 文档目的

这份文档用于把当前阶段的一个关键判断明确下来，供后续 AI 协作者、开发者和任务认领者参考。

核心结论只有一句话：**项目下一阶段的重点，不再是继续给地图叠加表现层，而是建立统一的“世界写回协议层”**，把玩家、地点、记忆、治理和时间压进同一套可演化、可约束的结构中。

---

## 一、当前阶段判断

从共享任务表看，浏览器主舞台与第一层体验闭环已经基本成立：

- `W1 / W1a / W2 / W3` 已完成：地图主舞台、世界地图骨架、平移缩放、悬停提示、侧边联动都已具备
- `V1 / V2 / V3` 已完成：视觉转义、扰动代理、治愈向精灵与情感锚点都已进入地图层
- `C1 / C2 / C3` 已完成：播报系统、记忆回声、镜像家园等参与感钩子已建立
- `D1 / D2` 已完成：语义缩放、玩家移动与街道通行抽象已进入主舞台

真正仍处于后续主线上游位置的，是以下任务：

- `D3`：玩家参与感与城市神话共创主线文档化收束
- `P3`：玩家写回权限与语义治理边界
- `P4`：历史深度 / 尘封之眼 / Time Folds 协议

因此，当前主矛盾已经发生变化：

> 现在的问题不是“地图能不能成立”，而是“玩家如何写入世界、世界如何记住玩家、这些变化如何被约束并长期演化”。

---

## 二、阶段重心应从空间层转向协议层

此前的推进重点是：

1. 让世界可看
2. 让地图可走
3. 让地点可点
4. 让气氛可感

而后续更值得优先建设的是：

1. 玩家行为如何抽象成统一事件
2. 这些事件如何影响玩家状态与地点状态
3. 这些变化如何决定世界广播、记忆回声、神话线索与家园反馈
4. 哪些变化只属于个人，哪些能进入局部公共世界，哪些能影响全局世界
5. 时间与历史如何叠加到同一地点之上

也就是说，下一阶段的核心不应是继续往地图上添加更多零散功能点，而应是先补齐“**状态—记忆—治理—时间**”的一致性协议。

---

## 三、玩家状态文档已经提供了统一骨架

[`docs/PLAYER_STATE.md`](docs/PLAYER_STATE.md) 实际上已经给出了后续系统最重要的状态基线。

这份文档把玩家状态拆成三层：

1. 即时状态层
2. 世界关系层
3. 长期人格 / 审美层

这意味着后面的共创、家园、阵营、历史回声、公共修复、命名权，不应该各自生长成互相割裂的新结构，而应优先挂接到已有状态轴上，例如：

- `current_position`
- `current_zone_id`
- `nearest_poi_id`
- `action_state`
- `clarity / tension / attunement / fatigue / warmth`
- `faction_standing.<faction_id>.*`
- `zone_familiarity.<zone_id>`
- `poi_familiarity.<poi_id>`
- `route_familiarity.<route_id>`
- `style_affinity.*`

建议后续新增功能时遵循一个原则：

> **先问“它改写哪条已有状态轴”，再问“它需要新增什么字段”。**

如果一个新玩法完全无法映射到现有状态结构，才考虑补新字段；否则优先复用已有状态语言，避免世界模型碎裂。

---

## 四、`D3` 应升级为“世界写回总入口设计”

当前 `D3` 的表述是“玩家参与感与城市神话共创主线文档化收束”，但从整个项目结构来看，这个任务更适合进一步提升定位。

建议把 `D3` 视为以下内容的总入口：

### 1. 定义玩家参与行为的最小事件模型

例如把以下行为抽象成统一世界事件：

- 移动
- 驻足
- 观察
- 发现
- 收集
- 留言
- 修复
- 命名
- 绑定记忆
- 触发扰动
- 参与神话线程

### 2. 定义事件到状态的更新链路

即：

`玩家事件 -> 玩家状态变化 -> 地点状态变化 -> 世界反馈`

这里的“世界反馈”应覆盖：

- 地图主舞台视觉反馈
- 播报系统文本反馈
- 地点回声与痕迹反馈
- 家园 / 阵营 / 神话线索反馈

### 3. 定义参与行为的可见性等级

例如：

- 只自己可见
- 区域内公共可见
- 需要系统精选后再公开
- 需要治理流程后才能成为世界常驻内容

### 4. 为 `P3` 提供上游输入

`D3` 不应只是叙事汇总文档。

更重要的是，它应该为 [`docs/AI_SHARED_TASKLIST.md`](docs/AI_SHARED_TASKLIST.md) 中的 `P3` 提供明确输入：

- 什么叫一次有效写回
- 哪些行为是即时反馈，哪些行为是持久写回
- 哪些字段可以被 AI 改写、压缩、重述、翻译
- 哪些内容必须保留原意或原始记录

换句话说，`D3` 最有价值的升级方向，是成为“**玩家如何进入世界写回系统**”的设计总入口。

---

## 五、`P3` 是当前最关键的防失控协议

随着参与感、留言、回声、命名和共创逐步进入系统，`P3` 的优先级已经非常高。

如果没有清晰的写回治理边界，项目会迅速退化成：

- 任意字符串堆积
- 地点氛围被噪声污染
- AI 改写无边界
- 公共世界质量失控
- “世界感”被留言板感取代

因此，建议把 `private / local_public / global` 三层写回，不只是定义为可见性，而是定义为三种不同的世界写回语义：

### 1. `private`

只影响玩家个人镜头与私人世界层：

- 私密留言
- 个人情绪胶囊
- 家园反馈
- 个人记忆回声
- 个体化的世界文案微调

### 2. `local_public`

可进入局部公共空间，但作用范围受限：

- 特定区域回声池
- 地点氛围句子
- 局部共创痕迹
- 小范围修复记录
- 特定社区或地点内部的轻量命名层

### 3. `global`

只能通过高门槛机制进入全局世界层：

- 系统任务完成后的地标变化
- 多人共识形成的神话命名
- 审美治理或投票通过的世界规则修改
- 被系统精选并长期保留的全城广播 / 城市传说

建议 AI 协作者在设计相关功能时，始终先判断：

> 这个内容究竟属于 `private`、`local_public`，还是 `global`？

如果这个问题还回答不了，就不应该急着落实现层。

---

## 六、`P4` 不只是历史系统，而是地图深度系统

[`docs/WEB_2D_SPIRIT_VIEW.md`](docs/WEB_2D_SPIRIT_VIEW.md) 已明确提出：风格切换应是世界镜头，而不是滤镜。

在这个前提下，`P4` 的价值不只是补一套“历史 lore”，而是让主舞台从“平面此刻”升级成“同一地点可被多层时间解释”的纵深容器。

也就是说，未来地图不只有：

- 空间层
- 风格层

还应有：

- 当下层
- 旧日回声层
- 尘封地标层
- 深层入口层
- 时间折叠层

这样同一个 POI 或 landmark，才可以既是现实地点，也是：

- 曾经发生过事件的证据容器
- 过去地标残影的入口
- 特定任务的历史回声触发器
- 某条神话线在旧时间层中的显现点

因此，建议把 `P4` 的设计目标改写为：

> **让地图从“横向空间结构”升级为“可叠加时间解释的世界深度结构”。**

---

## 七、建议的推进顺序

基于当前仓库状态，更推荐以下顺序，而不是直接扩写 `E1 / E2 / E3 / E4`：

### 第一优先级：收束 `D3`

目标：把玩家参与行为整理为统一事件模型与反馈链路。

产出建议：

- 玩家参与事件分类
- 事件字段草案
- 事件到状态更新链路
- 事件到世界反馈映射表

### 第二优先级：完成 `P3`

目标：明确世界写回权限、语义治理、AI 改写边界与内容审核口径。

产出建议：

- `private / local_public / global` 写回语义说明
- 每层允许写入的内容类型
- AI 改写 / 总结 / 翻译 / 内容审核的边界
- 可持久化与不可持久化内容划分

### 第三优先级：完成 `P4`

目标：定义时间折叠、历史证据、旧地标回声与深层入口协议。

产出建议：

- 历史证据来源类型
- 可信度与来源权重模型
- 时间层入口触发规则
- 地点跨时间表现结构

### 第四优先级：再扩展 `E1 / E2 / E3 / E4`

原因是这几个任务天然依赖上面的协议层：

- 精灵共同发现需要写回边界
- 公共修复任务需要公共世界持久化规则
- 命名权需要治理与可见性分层
- 玩家据点与幽灵回放需要记忆 / 时间 / 痕迹协议

换句话说，`E` 组任务应该建立在已成型的世界协议之上，而不应反过来逼协议临时拼凑。

## 八、建议补一份机器可读的玩家—世界事件结构

当前仓库已经存在规则型和 JSON 化的趋势，例如 [`docs/CAUSAL_RIPPLE_RULES_V0.1.json`](docs/CAUSAL_RIPPLE_RULES_V0.1.json)。

因此，下一步非常适合补一份机器可读的“玩家—世界事件结构”，作为浏览器层、命令行层、预览页、未来引擎层共享的统一输入结构。

建议至少覆盖以下事件族：

- `move`
- `idle`
- `observe`
- `discover`
- `collect`
- `leave_mark`
- `restore_site`
- `name_place`
- `bind_memory`
- `trigger_disturbance`
- `join_mythline`

建议基础字段包含：

- `event_id`
- `event_type`
- `player_id`
- `timestamp`
- `zone_id`
- `poi_id`
- `visibility_scope`
- `intent_tags`
- `state_delta`
- `world_delta`
- `moderation_state`
- `narrative_projection`

这类结构的价值在于：

1. 让后续系统共享同一套行为输入
2. 避免每个玩法单独发明字段
3. 为 AI 总结、改写、审核、回放提供统一中间层
4. 为未来 Godot / Canvas / Pixi 或服务端同步预留稳定接口

---

## 九、给 AI 协作者的行动原则

后续认领任务时，建议优先遵循以下原则：

### 原则 1：不要继续把地图当作单纯表现层

地图现在已经是世界主舞台。

任何新增功能，都应同时考虑：

- 它如何改变玩家状态
- 它如何改变地点状态
- 它如何形成世界记忆
- 它是否需要治理边界
- 它是否应进入时间层

### 原则 2：先协议，后功能扩写

当一个功能涉及：

- 写回
- 命名
- 共创
- 历史层
- 公共可见变化

优先先补设计文档或规则说明，再做具体实现。

### 原则 3：优先复用已有状态语言

新增设计应优先映射到：

- 玩家状态三层结构
- 世界地点现有语义
- 广播 / 回声 / 家园 / 神话线的既有反馈链路

而不是平行再造一套不可兼容的系统。

### 原则 4：公共世界变化必须有门槛

凡是会影响公共世界长期表意的内容，都不应直接裸写入。

至少应明确：

- 来源
- 权限层级
- AI 可改写范围
- 持久化条件
- 回滚或失效机制

### 原则 5：把“时间”当结构，不当装饰

历史与旧地标不只是补充 lore，而是后续地图深度的重要来源。

凡是设计地点回声、任务入口、记忆绑定、神话线程时，都应优先思考它是否可以兼容未来的时间层模型。

---

## 十、结论

对当前项目而言，下一阶段最值得投入的不是继续增加地图表层内容，而是建立统一的“世界写回协议层”。

这个协议层至少应同时回答五个问题：

1. 玩家做了什么，如何被表示成统一事件
2. 事件如何改写玩家状态与地点状态
3. 哪些变化只属于个人，哪些可以进入公共世界
4. AI 在其中可以如何改写、压缩、翻译与治理
5. 同一地点如何叠加时间、历史与尘封回声

只有这层成立，后续的神话共创、地点命名、精灵生态、公共修复、据点系统、幽灵回放和历史深潜，才会真正长在同一世界上，而不是变成一组互相脱节的功能集合。
