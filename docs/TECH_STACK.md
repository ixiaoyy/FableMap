# FableMap 技术栈与技术选型

## 总体原则

FableMap v0.1 的技术选型遵循三个原则：

1. **先验证闭环，不先追求极致性能**
2. **让数据层、语义层、表现层职责分离**
3. **优先选择生态成熟、开发速度快的工具**

## 推荐技术栈

### 数据与世界生成层

- **Python 3.11+**
- `requests`：调用 Overpass API
- `pydantic`：定义 Schema 和输入输出模型
- `typer` 或 `argparse`：构建 CLI
- `sqlite3`：本地持久化

### 表现层

- **Godot 4**
- GDScript（优先）或 C#（可选）

### 文档与样例

- Markdown
- `examples/` 中保存示例坐标与世界 JSON

## 为什么 v0.1 选择 Python

### 1. 它最适合数据管线原型

FableMap 的第一阶段本质上是一个“地理数据 -> 语义转换 -> JSON 输出”的数据管线问题。Python 在这类工作上开发速度极快，代码也更容易调整。

### 2. 它在 GIS / API / AI 工具链上生态成熟

Python 对以下场景都很友好：

- 请求 Overpass / HTTP API
- 处理 JSON / 地图数据
- 进行规则映射和文本模板化
- 接入后续的 LLM、向量库、H3、地理分析库

### 3. 它非常适合做 CLI 和离线生成器

第一版最重要的交付物是一个：

> 输入坐标，输出世界描述 JSON 的命令行工具

Python 做这类工具简单、直观、上手快。

### 4. 它能把“试错成本”压到最低

在项目早期，最大风险不是性能，而是：

- Schema 设计不稳
- 映射规则不对
- 生成链路不闭环

Python 非常适合快速迭代这些核心设计。

## 为什么不是一开始就用其他语言

### 不优先用 C++ / Rust

它们在性能和类型安全上更强，但 v0.1 的瓶颈不在这里。项目当前最需要的是快速验证世界生成逻辑，而不是极致优化。

### 不优先全用 C#

Godot 侧当然可以用 C#，但如果连数据管线、HTTP 请求、世界记忆和规则引擎都强绑定在引擎里，会让原型迭代变慢，也不利于 CLI 和批量生成。

### 不优先用 Node.js

Node 做 CLI 也可以，但 Python 在地图处理、数据建模、AI 生态和后续扩展性上更贴近 FableMap 的核心需求。

## 长期演化建议

FableMap 不必永远只用 Python。

推荐长期结构：

- **Python**：地理解析、世界生成、世界记忆、批处理工具
- **Godot / GDScript / C#**：场景加载、交互、实时表现
- **未来可选 Rust / Go**：高性能服务、地图切片处理、同步层

## 当前结论

对 FableMap v0.1 来说，Python 不是“最终唯一语言”，而是**最适合把项目从想法快速推到可运行原型的语言**。
