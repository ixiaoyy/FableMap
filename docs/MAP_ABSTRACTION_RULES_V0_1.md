# FableMap 地图抽象尺度与对象转译规则（历史参考）

## 文档定位

这份文档记录的是 FableMap 在“自绘地图 / Web-2D 主舞台”阶段形成的一套地图抽象思路。

它当时试图回答的问题是：

> 如何把真实城市从“现实底图”转成一个更像可探索世界地图的结构化空间？

但当前产品主线已经不再以这条路线为核心。

当前主线已经切换为：

> **真实底图 + 地点选择 + 角色遭遇 / 地点事件 + 聊天叙事 + writeback / memory**

因此，本文档现在只保留为**历史参考**，不再作为当前实现基线。

如需当前口径，请优先阅读：

- [`PRODUCT_BRIEF.md`](PRODUCT_BRIEF.md)
- [`ARCHITECTURE.md`](ARCHITECTURE.md)
- [`CURRENT_TASKS.md`](CURRENT_TASKS.md)
- [`INDEX.md`](INDEX.md)

---

## 它当时试图解决什么

在旧地图阶段，项目面临一个真实问题：

- 真实城市信息密度太高
- 如果一比一保留建筑、道路、店铺和 POI，地图会非常难读
- 如果完全不抽象，又很难承载事件、记忆、命名和写回

因此这份文档提出了一套旧时代的目标：

1. 保留现实关系
2. 主动压缩空间颗粒度
3. 让区域、节点、道路更像“可操作世界地图”
4. 让每个对象更适合承载状态、记忆和交互

这在“地图是主要舞台”的前提下是合理的。

---

## 它为什么不再是当前主线

后来这套抽象规则被降级，核心原因不是它完全错误，而是产品重心改变了。

### 1. 当前重点不再是“把地图画成世界地图”

现在真正优先的不是：

- 把建筑合并成据点
- 把街区压缩成区域单元
- 把现实地图做成 RPG 城镇图

而是：

- 让用户在真实地点上进入场景
- 让地点承载角色遭遇与地点事件
- 让聊天叙事成为主要体验舞台
- 让 writeback / memory 形成可回访关系

### 2. 地图抽象不能替代地点体验设计

旧文档主要解决“空间怎么抽象”，但当前更关键的问题是：

- 地点如何被选择
- 地点如何触发角色
- 地点如何进入聊天
- 事件如何写回世界
- 世界如何记住用户与地点关系

换句话说，旧文档偏重空间结构，而当前主线偏重体验结构。

### 3. 旧抽象规则绑定了旧前端地图链路

这份文档天然服务于以下历史链路：

- [`frontend/src/WorldMap.jsx`](../frontend/src/WorldMap.jsx:1)
- [`frontend/src/worldMap/geometry.js`](../frontend/src/worldMap/geometry.js:1)
- [`frontend/src/worldMap/renderers.js`](../frontend/src/worldMap/renderers.js:1)

而这些模块现在都已经被定义为冻结或历史参考，不再应该继续扩张为主线工程。

### 4. 真实底图接入降低了“自定义地图抽象”的必要性

既然当前方向已经接受：

- 直接使用真实底图作为空间容器
- 让地图承担定位、浏览、选点能力
- 把主要产品密度放在地点面板、事件层、角色层和聊天层

那么旧式“先抽象地图，再承载体验”的优先级自然下降。

## 这份文档现在还能提供什么价值

虽然它不再是当前基线，但仍有一些参考价值。

### 1. 旧数据收束思路参考

它提醒我们：真实城市对象过多时，确实需要某种“交互颗粒度收束”。

这个思想在当前方向下仍可能被迁移到：

- 地点聚合
- 地点卡片排序
- 邻近地点分组
- 事件触发范围控制

### 2. 世界对象命名与层级意识参考

它曾经强调：

- 区域层
- 节点层
- 气氛层

这些层级意识虽然不应再驱动自绘地图，但仍可转化为：

- base map 层
- place graph 层
- character / event 层
- chat narrative 层

### 3. 历史决策背景参考

它有助于后续协作者理解：为什么项目里会出现大量关于“区域单元”“节点抽象”“地图颗粒度”的早期讨论。

## 当前处理口径

### 可以保留为参考的部分

- 现实对象不应被原样堆满界面的判断
- 用户体验需要颗粒度控制的意识
- 交互对象应该少而有意义的原则

### 不再作为当前主线推进的部分

- 以“世界地图化抽象”作为近期核心工作
- 围绕自绘地图继续设计区域 / 节点 / 气氛分层
- 为地图可读性继续投入大量专门抽象规则
- 把旧地图抽象协议作为当前前端重构中心
