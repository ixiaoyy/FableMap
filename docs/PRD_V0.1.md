# FableMap PRD v0.1

## 版本目标

构建一个最小可运行原型，证明以下三件事成立：

1. 真实地图数据可以被稳定翻译为幻想世界语义
2. 同一地点的生成内容可以保持一致并具备演进能力
3. 生成结果可以被 Godot 消费并转为可探索场景

## 产品方向说明

v0.1 仍然只解决“世界生成闭环”，但项目长期方向会走向更面向成年人与硬核玩家的体验，包括：

- 区域阵营与权力博弈
- 现实生活的黑色幽默与社会讽刺
- 基于真实地点历史的谜团探索
- 由现实动态信号驱动的世界扰动

这些能力不进入 v0.1 交付范围，但会影响现在的 Schema 和架构预留。

## 范围定义

### In Scope

- 输入经纬度坐标
- 调用 Overpass API 拉取周边 OSM 数据
- 提取建筑、道路、绿地、商业、医疗、教育、工业、水体等要素
- 将 OSM 数据映射为统一的世界描述 JSON
- 为区域分配稳定 ID
- 将结果持久化到本地存储
- 在 Godot 中加载 JSON 并生成基础场景
- 点击 POI 查看名称、描述和标签

### Out of Scope

- AR 摄像头叠加
- 多人联机
- 战斗系统
- 复杂经济系统
- 大规模在线世界同步
- 完整任务链图数据库
- 高精度美术资产生产

### Strategic Later Scope

- 阵营声望与控制力系统
- 区域消费/资源驱动的虚拟经济层
- NPC 欺骗、收买、办公室政治等复杂人性模拟
- 基于历史事件碎片的谜团链路
- 基于人流/拥堵/营业状态的动态地区扰动

## 目标用户行为

1. 用户执行 CLI 命令并输入坐标
2. 系统生成该区域的世界 JSON
3. 用户查看生成结果或把结果传给 Godot Demo
4. 用户在场景中移动并点击 POI 查看异世界描述
5. 再次生成同一区域时，核心结构和命名保持稳定

## 功能需求

### FR-1 坐标输入

- 系统必须支持 `lat` / `lon` 输入
- 初期不强制支持地名解析

### FR-2 地图数据获取

- 系统必须能按坐标半径请求 Overpass API
- 请求失败时需要有明确错误输出

### FR-3 OSM 语义转换

- 系统必须把 OSM 原始标签转成统一中间结构
- 中间结构必须支持区域级和 POI 级信息

### FR-4 幻想世界映射

- 系统必须为区域生成 `theme`、`atmosphere`、`faction_hint` 等字段
- 系统必须为 POI 生成 `fantasy_name`、`fantasy_type`、`description`

### FR-5 世界记忆

- 系统必须为区域生成稳定 key
- 同区域重复生成时应优先复用历史记录

### FR-6 场景输出

- 系统必须输出 Godot 可读取的 JSON
- JSON 必须包含对象类型、位置、视觉提示与描述文本

## 非功能需求

- **一致性**：同一区域的核心语义不能随机飘移
- **可解释性**：每个 POI 的映射应能追溯回原始 OSM 类型
- **可扩展性**：后续可以增加更多标签规则与视觉预制体
- **离线可重放**：已生成区域应可本地再次加载

## 成功标准

- 能稳定生成至少一个城市街区的幻想世界 JSON
- 对同一坐标重复运行后，关键地点命名与类别稳定
- Godot Demo 能渲染至少三类 POI 和两类区域风格
- 项目文档清晰到足以支持后续工程实现
