# FableMap Web-2D 灵觉视图设计说明（历史参考）

## 文档定位

这份文档记录的是 FableMap 曾经把“浏览器自绘 2D 地图主舞台”作为核心产品方向时的设计判断。

它**只保留为历史参考**，用于解释项目为什么曾经走向 [`frontend/src/WorldMap.jsx`](../frontend/src/WorldMap.jsx:1) 这条链路，以及为什么这条链路后来被降级。

当前主线已经切换为：

> **真实底图 + 地点选择 + 角色遭遇 / 地点事件 + 聊天叙事 + writeback / memory**

因此本文档中的“主地图承载空间”“Web-2D 主舞台优先”“地图作为核心体验容器”等判断，**不再代表当前产品方向**。

如需当前口径，请优先阅读：

- [`PRODUCT_BRIEF.md`](PRODUCT_BRIEF.md)
- [`ARCHITECTURE.md`](ARCHITECTURE.md)
- [`CURRENT_TASKS.md`](CURRENT_TASKS.md)
- [`INDEX.md`](INDEX.md)

---

## 它当时试图解决什么

在项目早期，团队希望先在浏览器中验证：

1. 真实地图骨架能否被转译成一个可阅读、可点击、可探索的幻想世界
2. 用户能否通过 2D 主舞台而不是纯文本卡片进入地点体验
3. Web 是否可以充当比 Godot 更轻量的第一验证舞台

这份文档正是在那个阶段形成的。

---

## 它为什么曾经合理

当时做 Web-2D 的原因主要有：

- 复用早期 `nearby -> page -> bundle` 的浏览器链路
- 用标准 Web 技术快速验证“现实地点 -> 世界化表达”
- 避开 3D、美术资产、引擎整合带来的前置成本
- 为未来的视觉探索准备一个可快速试错的平面舞台

这些判断在原型阶段是成立的。

---

## 它后来为什么被降级

随着工程继续推进，旧 Web-2D 主舞台暴露出几个根本问题：

1. **地图不是体验目标本身，只是容器**
   - 用户真正关心的是地点里的人、事件、故事与可持续关系
   - 把大量精力投入自绘地图，只会持续放大非核心复杂度

2. **AI 自动生成地图视觉的质量与一致性不足**
   - 地图资产、图标包、场景绘制很难稳定支撑长期产品体验
   - 视觉炫技无法替代持续叙事、角色关系与地点记忆

3. **前端旧链路耦合过重**
   - [`frontend/src/WorldMap.jsx`](../frontend/src/WorldMap.jsx:1) 同时承担布局推导、命中、绘制、资源预载等职责
   - [`frontend/src/worldMap/renderers.js`](../frontend/src/worldMap/renderers.js:1) 与 [`frontend/src/worldMap/geometry.js`](../frontend/src/worldMap/geometry.js:1) 主要解决“如何画地图”，而不是“如何生成地点叙事体验”

4. **产品重心已经改变**
   - 新主线不再是“把现实地图画成异世界地图”
   - 新主线是“把真实地点变成可进入的聊天场景、角色遭遇与事件节点”

## 这份文档现在还能提供什么价值

虽然它不再指导当前实现，但仍保留三类参考价值：

### 1. 历史判断参考

它解释了项目为什么会形成浏览器 2D 地图阶段，以及为什么旧前端地图链路存在。

### 2. 视觉语言参考

其中关于氛围、镜头、情绪层与现实信号转译的想法，仍可被迁移到：

- 地点详情面板
- 角色遭遇氛围
- 事件卡片
- 聊天场景的语气与世界反馈

### 3. 降级边界参考

它帮助团队识别哪些内容应继续保留为“增强层灵感”，而不是重新变回“产品主舞台”：

- 风格模式
- 视觉氛围层
- 情绪广播
- 世界镜头切换

## 当前处理口径

基于当前重构方向，这份文档中的内容应这样理解：

### 保留为灵感的部分

- 现实地点与世界气氛之间的转译思路
- 不同镜头下的叙事语气差异

- 为地图视觉成立而继续扩张渲染层
- 以地图资产包作为近期主工程方向
