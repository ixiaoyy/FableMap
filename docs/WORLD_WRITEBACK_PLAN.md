# FableMap 世界写回协议推进计划 v0.1

## 文档目的

这份文档用于把下一阶段的工作重点，从“继续给地图叠加表现层”，正式收束到“建立统一世界写回协议层”。

它不直接替代详细协议正文，也不是最终 API 文档。

它的作用是：

1. 明确为什么下一阶段应优先做世界写回协议
2. 把实施顺序拆成可执行、可验收的阶段任务
3. 为后续 `P3` / `P4`、后端接口、前端交互与数据持久化提供统一推进路线

---

## 一句话结论

下一阶段最合适的优先级是：

1. **先做统一事件模型**
2. **再打通最小写回闭环**
3. **再补可见性与治理边界**
4. **最后再扩历史深度 / Time Folds**

换句话说：

> 先回答“玩家做了什么、地点怎么变、世界记住了什么”，再回答“这些变化谁能看到、能保留多久、如何跨时间沉淀”。

---

## 为什么现在优先做这个

根据当前仓库状态，下面这些层已经基本成立：

- Python 世界生成闭环已经存在
- 浏览器 2D 主舞台已经成立
- 地点详情、移动、缩放、视觉代理、记忆回声、镜像家园等前置层已经存在
- FastAPI + 前端框架化结构已经接入
- `D3` 已经把“玩家如何进入城市神话”整理成了展示与叙事层入口

因此当前的主要瓶颈已经不是：

- 地图还能不能再多一点表现
- 页面还能不能再多一点视觉层

而是：

- 玩家行为能不能进入统一事件流
- 地点状态能不能被真实改写
- 世界反馈能不能被稳定复用
- 写回结果能不能被持久化并在下次进入时继续存在
- 不同层级的写回能不能被治理与约束

这意味着，项目当前最稀缺的已经不是展示能力，而是**演化能力**。

---

## 阶段目标

本轮计划的总目标是：

> 让浏览器里的玩家第一次能够对世界切片产生最小、稳定、可重复进入的写回结果。

对应的里程碑定义为：

## 里程碑：Persistent Slice v0.1

完成标准：

1. 玩家可在 Web 地图中选中一个地点或所在区域
2. 玩家可执行最小写回动作：`observe / dwell / mark`
3. 行为会通过 API 以统一事件结构提交到后端
4. 后端会生成统一事件记录与状态变化结果
5. 世界切片中至少一部分状态可被持久化
6. 再次打开同一切片时，仍能看到熟悉度、标记、回声等痕迹

一旦这个里程碑成立，项目就从“可交互地图原型”进入“可持续世界原型”。

---

## 阶段一：统一世界写回协议骨架

### 目标

先建立统一语言，避免后续前端、后端、文档、状态结构各自生长。

### 本阶段要解决的问题

1. 玩家行为如何被命名
2. 行为作用到谁
3. 行为带来哪些状态变化
4. 状态变化属于什么可见范围

### 建议输出物

新增正式协议文档：

- `docs/WORLD_WRITEBACK_PROTOCOL.md`

### 协议最小结构建议

#### 1. event schema

定义统一事件结构，至少包含：

- `event_id`
- `event_type`
- `player_id`
- `timestamp`
- `target`
- `payload`
- `visibility`
- `source`

建议第一版事件类型先限制在：

- `observe`
- `dwell`
- `mark`

后续再扩：

- `collect`
- `bind_memory`
- `leave_echo`
- `repair`
- `name_place`

#### 2. target schema

定义事件的作用目标，建议支持：

- `poi`
- `zone`
- `route`
- `home`
- `world`

最小字段建议：

- `target_type`
- `target_id`
- `slice_id`
- `coordinates`（可选）

#### 3. effect schema

定义一次事件可能引发的状态变化，建议分为三层：

- `player_effects`
- `place_effects`
- `world_effects`

示例：

- 玩家 `attunement +1`
- 玩家 `clarity +1`
- 玩家 `tension -1`
- `poi_familiarity +1`
- `zone_familiarity +1`
- 新增一条 `place_mark`
- 新增一条 `local_echo`
- 追加一条 `broadcast_hint`

#### 4. visibility schema

建议写回可见性从第一版就进入协议，而不是事后补规则。

最小级别：

- `private`
- `local_public`
- `global`

### 阶段验收标准

- 协议文档能明确描述 `event / target / effect / visibility` 四层结构
- `observe / dwell / mark` 三种行为具备统一命名
- 能和当前 [`docs/PLAYER_STATE.md`](docs/PLAYER_STATE.md) 的状态轴对接
- 能和后续 `P3` / `P4` 明确衔接

## 阶段二：最小写回闭环 MVP

### 目标

不要先做完整共创系统，只做一次真正可运行的写回闭环。

### 建议优先实现的三个行为

#### 1. `observe`

触发方式：

- 玩家选中 `poi` 后点击“观察”

建议结果：

- `action_state -> observing`
- `attunement +1`
- `poi_familiarity.<poi_id> +1`
- 该地点增加一条轻量 session 回声或 reveal 记录
- 地点详情可解锁更多解释字段

#### 2. `dwell`

触发方式：

- 玩家在区域停留数秒或主动触发“驻足”

建议结果：

- `action_state -> idle` 或 `observing`
- `clarity +1`
- `tension -1`
- `zone_familiarity.<zone_id> +1`
- 有概率生成 `local_echo`

#### 3. `mark`

触发方式：

- 玩家为地点添加一个轻量标签

建议第一版标签池限制为固定选项，例如：

- `safe`
- `uncanny`
- `warm_corner`
- `return_again`
- `rain_friendly`

建议结果：

- 新建一条 `place_mark`
- 默认 `visibility = private`
- 地点详情中出现“你的标记”
- 镜像家园或后续收藏系统可引用这些标记

### 本阶段建议交付物

#### 后端

- 一个最小事件处理器模块，例如：
  - `fablemap/writeback.py`
  - 或 `fablemap/web/service.py` 内先落最小处理链
- 一个最小持久化结构，用于保存：
  - 玩家状态快照
  - slice 级别熟悉度
  - POI 标记
  - 最近回声

#### API

新增最小接口，例如：

- `POST /api/world/event`

输入：统一事件结构

输出至少包含：

- `player_state`
- `place_state`
- `world_feedback`
- `event_record`

#### 前端

只增加三个动作入口：

- 观察
- 驻足
- 标记

不在本阶段扩更多复杂玩法。

### 阶段验收标准

- 三种行为都能从前端触发并通过 API 提交
- 后端能返回统一结构的变化结果
- 再次读取同一 slice 时，至少一部分写回状态仍然存在
- 前端能把“状态变化”显示出来，而不是只显示一次性提示文案

## 阶段三：治理边界与语义写回分层

### 目标

在最小闭环跑通后，立刻补 `P3`，避免写回失控。

### 本阶段要回答的问题

1. 什么内容只属于玩家自己
2. 什么内容可以进入局部公共空间
3. 什么内容可能进入全局神话层
4. 哪些内容允许 AI 改写、压缩、翻译或摘要
5. 哪些内容必须保留原始语义或原始记录

### 建议治理分层

#### `private`

适合：

- 私密记忆
- 个人标签
- 家园引用
- 个人情绪痕迹
- 个体熟悉度与滤镜偏好

#### `local_public`

适合：

- 某区域共享的地点传闻
- 局部公共标签
- 区域回声池
- 小范围修复痕迹
- 社区化的轻量共创句子

#### `global`

适合：

- 长期保留的城市神话命名
- 高门槛的全城广播
- 世界规则层更新
- 跨区域共识后的正式变更

### 本阶段建议输出物

- 在 `docs/WORLD_WRITEBACK_PROTOCOL.md` 增补治理章节
- 或新增：
  - `docs/WORLD_WRITEBACK_GOVERNANCE.md`

### 阶段验收标准

- `private / local_public / global` 的差异被协议化定义
- 至少能说明三种行为的默认 visibility
- 至少定义一套最小 AI 改写边界
- 为后续 `E1` ~ `E4` 提供可延续的治理接口

---

## 阶段四：Time Folds / 历史深度最小接入

### 目标

在写回与治理稳定之后，再让地点拥有时间纵深。

### 为什么不优先做

如果当前连“什么算一次有效写回”都没有定清，历史层会直接把复杂度放大。

所以更合理的顺序是：

1. 先定义事件
2. 再定义写回
3. 再定义可见性与治理
4. 最后定义时间如何沉淀这些变化

### 第一版历史层建议

先不要做复杂时间旅行，而是先让地点具备三层时间切片：

- `present_state`
- `recent_echoes`
- `archived_layers`

分别表示：

- 当前状态
- 最近发生的可追踪变化
- 被沉淀下来的历史痕迹

### 阶段验收标准

- 至少一个地点支持时间切片结构
- 玩家行为留下的部分痕迹能从 `recent_echoes` 进入 `archived_layers`
- 地图或详情面板能看到“过去发生过什么”的最小入口

---

## 数据结构推进顺序建议

### 第一层：文件级持久化

为快速验证闭环，第一版不必急着引入数据库。

可先使用：

- slice 级 JSON
- player state JSON
- event log JSON

优势：

- 开发快
- 可调试
- 可复现
- 与当前原型工程风格一致

### 第二层：状态层抽象

当最小闭环稳定后，再把持久化抽象为统一状态层，便于未来切到：

- SQLite
- PostgreSQL

### 第三层：正式数据库

当进入多人并发、公共写回、治理与历史沉淀阶段时，再迁移到数据库承担：

- 玩家状态
- 事件记录
- 地点状态
- 区域状态
- 公共写回与治理数据

核心原则：

> 先验证协议，再升级存储；不要反过来先做复杂存储，再回头寻找协议意义。

## 推荐的工程拆分

### 文档层

1. `docs/WORLD_WRITEBACK_PROTOCOL.md`
2. 视情况新增 `docs/WORLD_WRITEBACK_GOVERNANCE.md`
3. 在 `docs/AI_SHARED_TASKLIST.md` 中补充阶段任务状态与顺序

### 后端层

1. 定义统一事件输入结构
2. 定义事件 -> effect 的处理链
3. 定义最小状态持久化读写
4. 暴露 `POST /api/world/event`

### 前端层

1. 地点详情卡中加入 `observe`
2. 玩家状态卡或地图层加入 `dwell`
3. 地点详情卡中加入 `mark`
4. 将 API 返回的状态变化回写到 UI

### 测试层

1. 事件处理器单测
2. API 路由测试
3. 写回持久化测试
4. 前端最小交互链路测试或集成验证

## 两周范围建议

如果按短周期推进，建议把这轮工作收敛为下面的 Sprint 目标：

## Sprint 目标

> 让浏览器里的玩家第一次“写回世界”，并能在下一次进入同一世界切片时看到痕迹仍然存在。

### 第一周

- 完成 `WORLD_WRITEBACK_PROTOCOL v0.1` 文档
- 确定 `observe / dwell / mark` 事件模型
- 确定 target / effect / visibility 结构
- 实现后端最小事件处理器
- 打通 `POST /api/world/event`

### 第二周

- 前端接入三种最小写回动作
- 将状态变化显示到玩家状态 / 地点详情 / 回声面板
- 实现最小文件级持久化
- 补充测试与演示路径
- 验证重新进入 slice 后写回结果仍可见

---

## 当前不建议优先推进的事项

在本计划完成前，建议不要把重心放在以下方向：

- 新增更多地图表现花样
- 扩大量 POI 图标类型
- 更复杂 hover 与动画层
- 提前做复杂战斗系统
- 提前做完整社交平台化能力
- 提前做完整历史世界模拟

这些方向并非不重要，而是不应在当前阶段压过写回协议本身。

---

## 最终判断

FableMap 当前已经拥有地图的骨架、视觉的皮肤与初步叙事的呼吸。

下一步最重要的，不是继续让地图“更像一个世界”，而是让世界第一次能够因为玩家而发生真实变化。

因此，本轮最核心的工程任务不是增加新表现层，而是完成：

> **统一世界写回协议 + 最小写回闭环 + 可见性治理入口**

只要这条线跑通，后续的神话共创、公共修复、地点命名、轻社区、时间褶皱与历史沉淀，都会拥有统一、可持续扩展的地基。
