# FableMap 下一阶段技术优先级决策

## 文档目标

把 FableMap 当前的技术判断落成一个明确的执行结论：

1. 现在最优先做什么
2. 前后端分离是否应该立刻开始
3. 是否应该引入前端框架
4. 后端与数据层接下来如何演化
5. 什么事情现在不应该优先做

---

## 一句话结论

**当前最优先的不是多端并行，不是小程序，不是 App，而是把现有 Python 世界引擎升级为稳定的接口层，并引入一个前端框架完成轻量前后端分离。**

换句话说，下一阶段的主线应该是：

**世界生成核心继续留在 Python，前端进入框架化，前后端通过稳定 JSON 协议协作。**

---

## 当前项目状态判断

从现有结构看，FableMap 已经具备几条很重要的基础能力：

- [`fablemap/cli.py`](../fablemap/cli.py) 提供 CLI 入口
- [`fablemap/world_builder.py`](../fablemap/world_builder.py) 已经承载世界生成逻辑
- [`fablemap/cache.py`](../fablemap/cache.py) 提供请求缓存
- [`fablemap/page.py`](../fablemap/page.py) 已经有本地页面驱动预览能力
- [`fablemap/bundle.py`](../fablemap/bundle.py) 可以导出静态 bundle
- [`demo-output/bundle/index.html`](../demo-output/bundle/index.html) 说明浏览器展示链路已经成立

这意味着项目已经不再是“纯想法”阶段，而是进入了**需要稳定边界、整理结构、为后续交互复杂度做准备**的阶段。

当前最值得解决的，不是“能不能多端发”，而是下面三个问题：

1. 世界生成能力是否能被稳定复用
2. 前端交互复杂度上升后是否还能持续维护
3. 数据状态是否有机会从文件产物逐步升级到正式状态层

---

## 最高优先级：轻量前后端分离

## 为什么它是现在的第一优先级

当前项目的核心逻辑已经天然分成两类：

### 1. 后端职责

- 拉取地理数据
- 做语义映射
- 生成世界 JSON
- 管理缓存
- 未来承接世界状态、记忆、任务、事件

这些能力都更适合继续保留在 [`Python`](./TECH_STACK.md:15) 侧。

### 2. 前端职责

- 地图舞台渲染
- 面板切换
- 玩家交互状态
- 动画与视觉反馈
- 多视图切换
- 移动端适配

这些事情用单页式、组件化前端来承接会更稳定。

如果继续把展示层和世界逻辑混在同一层里推进，后面会出现几个问题：

- 页面逻辑膨胀
- 状态越来越难管理
- 交互代码和数据逻辑纠缠
- 后续扩展移动 Web、小程序、App 时缺少清晰边界

所以当前最优先的动作不是“换语言”，而是**先把边界切开**。

---

## 第二优先级：引入前端框架

## 是否应该上框架

**应该。优先级很高。**

但这里的目的不是为了追流行，而是为了解决真实问题：

- UI 状态会越来越复杂
- 页面模块会越来越多
- 需要组件复用
- 需要统一路由、数据请求、状态同步方式
- 需要更自然地做响应式和移动端适配

### 推荐方向

优先建议：

- **后端**：Python API
- **前端**：Vue 3

原因：

- 学习和改造成本相对低
- 对地图面板、信息卡、弹层、侧边栏这类中型交互很合适
- 以后如果扩展到 H5，迁移思路自然
- 对当前阶段的小团队更友好

如果后续团队更偏工程化、组件规范化、全栈 React 生态，也可以选择 React，但对当前阶段来说，Vue 3 往往更轻、更顺。

---

## 第三优先级：把 Python 从脚本层升级到 API 层

这一步不是推翻现有 [`fablemap`](../fablemap) 目录，而是把已有能力重新组织。

### 当前保留

继续保留：

- CLI 能力
- 静态导出能力
- 本地缓存能力
- 世界生成规则能力

### 新增能力

新增一个 API 层，负责：

- 接收前端请求
- 调用世界生成逻辑
- 输出标准 JSON
- 暴露世界详情、区域详情、POI 详情等接口
- 为未来的用户状态与世界状态更新预留入口

推荐形式：

- Python + FastAPI

### 为什么不是立刻换后端语言

因为现阶段最核心的价值仍然在：

- 规则映射
- 世界生成
- 数据处理
- AI 接入潜力

这些都仍然更适合继续由 Python 承担，见 [`docs/TECH_STACK.md`](./TECH_STACK.md) 和 [`docs/ARCHITECTURE.md`](./ARCHITECTURE.md)。

---

## 第四优先级：数据层从“文件产物”升级为“数据库 + 文件并存”

现在的存储方式本质上还是：

- 请求缓存 JSON
- 世界输出 JSON
- bundle 静态文件

这对于原型验证是合理的，但对于下一阶段不够。

### 建议的数据层策略

不要立刻废弃 JSON 文件，而是升级为双层结构：

#### 1. 数据库负责状态

优先存这些：

- 生成任务记录
- 世界元信息
- 区域状态
- POI 状态
- 玩家状态
- 事件记录

#### 2. 文件负责产物与快照

继续保留这些：

- world 导出文件
- bundle 预览文件
- 调试快照
- 对外分享产物

### 推荐数据库

下一阶段优先推荐：

- 早期可用 SQLite 快速起步
- 更稳妥的产品化方向是 PostgreSQL

核心原则是：

**数据库存可查询、可更新、可并发的数据；JSON 文件存可导出、可复现、可演示的结果。**

---

## 当前不应该优先做的事情

以下事情都可以做，但不应该排在当前最高优先级：

### 1. 不应该优先做全端并行

当前不建议同时推进：

- Web
- H5
- 小程序
- App

正确顺序应是：

1. 先把 Web 主链打透
2. Web 兼容移动端，顺带覆盖 H5
3. 再根据渠道价值判断小程序
4. 最后再判断 App

### 2. 不应该优先大换后端语言

现在没有证据表明 [`Python`](./TECH_STACK.md:15) 已经成为瓶颈。

真正的瓶颈仍然会更可能出现在：

- 协议不稳定
- 交互结构不清晰
- 数据状态缺乏正式模型
- 展示层组织方式不足

### 3. 不应该优先上重微服务

当前阶段更适合：

- 单体核心
- 明确模块边界
- 统一协议
- 按热点逐步拆分

而不是一开始就进入复杂服务治理。

---

## 推荐执行顺序

## 阶段 A：先做边界整理

目标：把“Python 世界引擎”和“前端交互层”解耦。

建议动作：

1. 明确世界 JSON 的对外协议
2. 明确哪些能力属于 API，哪些能力属于导出工具
3. 新增 Python API 服务入口
4. 保留 CLI 与 bundle 作为调试和导出通道

交付标准：

- 前端不直接依赖内部脚本结构
- 后端有稳定 API 出口
- 同一份世界协议可被页面、导出、测试复用

## 阶段 B：前端框架化

目标：把当前浏览器预览升级成真正可扩展的交互层。

建议动作：

1. 选定 Vue 3 或 React 其一
2. 建立页面路由、组件层、状态层
3. 把世界地图、详情面板、情绪电台、调试面板模块化
4. 统一数据请求方式

交付标准：

- 前端页面不再依赖单文件拼接式结构
- 交互状态可维护
- 移动端适配策略可持续演进

## 阶段 C：数据层升级

目标：让世界状态和玩家状态具备正式持久化能力。

建议动作：

1. 为生成任务与世界元信息建立数据库表
2. 为区域状态、POI 状态建立正式模型
3. 保留 JSON 导出作为快照层
4. 逐步把页面查询从文件访问切到 API 查询

交付标准：

- 支持增量更新
- 支持按区域和 POI 查询
- 支持玩家状态回写

---

## 未来语言演化判断

当前建议保持：

- **Python**：继续作为核心世界引擎、规则层、API 编排层
- **前端框架**：承担交互与表现层

更后期如果出现明确瓶颈，再考虑：

- **Go**：高并发服务、网关、实时服务
- **Rust**：高性能空间计算、地图处理、底层引擎模块

原则不是“后面一定替换 Python”，而是：

**谁出现瓶颈，谁被独立优化。**

---

## 最终决策

### 当前最优先事项

**是的，当前最优先的方向就是：前后端分离，并在前端引入框架。**

但更准确地说，应该理解为：

1. **先做轻量前后端分离**
2. **前端进入框架化**
3. **后端继续用 Python，并新增 API 层**
4. **数据层开始从纯文件存储演化为数据库 + 文件并存**

### 建议的下一阶段标准技术组合

- **后端**：Python + FastAPI
- **前端**：Vue 3
- **数据层**：PostgreSQL（早期可先 SQLite）
- **缓存层**：保留现有 JSON 缓存，后续补 Redis
- **导出层**：继续保留 bundle / world JSON 产物

---

## 一句话路线图

**先把 Python 世界引擎 API 化，再把浏览器端框架化，然后再补正式数据库；多端、小程序、App 都放在这条主链稳定之后。**
