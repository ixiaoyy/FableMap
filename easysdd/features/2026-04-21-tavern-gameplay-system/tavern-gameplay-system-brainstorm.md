# AI 导演的酒馆玩法系统 Brainstorm

> Stage 0 | 2026-04-21 | 下一步：design

## 想做什么、为什么

本轮目标不是再加几个固定小游戏按钮，而是把酒馆从“聊天 + 模板开局”推进到“店主可以经营自己的可玩体验”。用户明确选择了完整玩法系统：店主可以添加任何玩法，默认公益酒馆也要按照各自主题增加贴合内容的专属玩法和剧情。

探索中形成的核心判断是：底层需要支持节点分支和结构化局内状态，否则玩法无法恢复、结算和验证；但店主侧不能被迫编辑复杂流程图。正确方向应是“AI 导演的轻配置玩法系统”：店主填玩法目标、氛围、素材、禁止事项和奖励文案，系统内部用 GameplayDefinition / GameplayNode / GameplaySession / GameplayEvent 承载；有 AI 时由 AI 根据当前状态决定下一步，没有 AI 时由默认节点选项和随机事件池兜底。

## 考虑过的方向

### 方向 A：轻量玩法卡

- 描述：复用现有聊天模板或小游戏模板，店主填写标题、简介、开局提示和剧情钩子，访客点击后发送普通聊天消息。
- 价值：实现快，协议风险低，能马上补足“店主可添加玩法”。
- 代价：没有结构化进度、恢复、完成状态和结算；更像高级快捷句，不是完整玩法系统。
- 结论：否决为主方案。它可以作为完整系统里的“简单创建模式”，但不能满足“功能完整可用”。

### 方向 B：完整节点分支玩法系统

- 描述：定义玩法、节点、选项、会话和事件；访客开始一局后有可恢复状态，AI 或规则兜底推进节点。
- 价值：能支持专属剧情、局内进度、完成状态、文字奖励、回访延续和测试验证。
- 代价：需要 Schema / API / UI / 默认数据 / 测试同步，属于高风险协议变更。
- 结论：选定。用户明确表示“不着急快速推出第一版，第一版要求功能完整可用”。

### 方向 C：AI 自由主持 + 状态快照

- 描述：店主写规则，AI 自由决定推进；系统只保存当前摘要、完成状态和最近玩法 ID。
- 价值：配置最轻，AI 表现空间大。
- 代价：难以测试、难以回放、无 AI 时兜底弱，容易变回纯聊天。
- 结论：部分吸收。AI 可以当导演决定下一步，但必须落到结构化 GameplaySession / GameplayEvent 上。

### 方向 D：脚本化游戏引擎

- 描述：店主或高级用户编写可执行规则、条件表达式、复杂状态机或脚本。
- 价值：表达力最强。
- 代价：安全和复杂度高，偏离赛博酒馆；容易走向战斗、等级、装备和传统 RPG 系统。
- 结论：否决。首版不允许可执行脚本和复杂游戏引擎。

## 选定方向与遗留问题

选定方向是 **AI 导演的轻配置完整玩法系统**。

核心用户行为：

1. 店主在后台创建玩法，只填少量表单：玩法名称、目标、简介、氛围、素材、禁止事项、结局/奖励文案。
2. 系统把轻配置归一化为结构化玩法定义，必要时生成默认节点和随机事件池。
3. 访客进入酒馆后看到贴合该酒馆的玩法入口，选择一个玩法开始。
4. 系统创建 GameplaySession，保存当前节点、状态、事件日志和完成情况。
5. 有 AI 时，AI 根据玩法定义、酒馆设定、NPC 和会话状态决定下一步；没有 AI 时，系统按默认选项或随机事件池推进。
6. 完成后写入会话结算，可用于回访反馈；是否写入长期记忆留给 design 明确。

明显不做：

- 不做战斗、等级装备、竞技排行榜、传统 RPG 任务系统。
- 不做访客间实时社交或跨酒馆玩家社交。
- 不允许店主上传可执行脚本、JS 代码或任意规则表达式。
- 不让平台自动替普通店主发布酒馆内容；若后续有 AI 辅助生成，也必须是店主确认后的候选草稿。
- 不把地图玩法化为路线规划、打卡导航或传统地图 App 功能。

留给 design 的问题：

- GameplayDefinition / GameplayNode / GameplaySession / GameplayEvent 的最小字段和持久化位置。
- AI 导演返回结构化推进结果的契约，以及无 AI 时随机事件兜底的确定性测试方式。
- 店主“轻配置”如何映射到底层节点，避免 UI 复杂。
- 默认 4 个公益酒馆分别预置哪些玩法，如何保证公益、安全和主题贴合。
- 玩法完成后是否生成 MemoryAtom，还是只写 GameplaySession 结算。
- 酒馆包导入导出是否包含玩法定义，以及必须排除哪些访客会话数据。

## 仓库扫描备注

- 现有 `frontend/src/tavernMiniGames.js` 和 `TavernMiniGamePanel.jsx` 已提供 AI 主持小游戏模板，但只是普通聊天开局，不保存局内结构化状态。
- `fablemap/tavern.py` 已有 `chat_templates` 元数据桶，但它语义偏聊天模板；完整玩法系统应避免复用这个名字承载 GameplaySession。
- 默认公益酒馆位于 `fablemap/default_taverns.py`，当前有 4 个主题明确的公益酒馆，适合预置专属玩法。
- `docs/WHAT_NOT_TO_BUILD.md` 明确排除战斗、等级装备、排行榜、传统 RPG、访客间社交、平台生成酒馆内容；本方向必须持续守住这些边界。
- `easysdd/compound/` 当前无相关沉淀文档；已有 feature design 检索未命中完整 gameplay 系统。
